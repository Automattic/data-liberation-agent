# post_content conventions

What's legal in `post_content` block markup, what isn't, and why.

## Hard rules — output is rejected if any fail

1. **Round-trip through `parse_blocks()`.** Every `<!-- wp:foo -->` opens has a matching `<!-- /wp:foo -->` close, OR is self-closed (`<!-- wp:foo /-->`). Open/close pairs must nest correctly — `<!-- wp:a --> <!-- wp:b --> <!-- /wp:a --> <!-- /wp:b -->` is rejected.

2. **Only allow-list block types.** Restricted to:
   - `core/paragraph`
   - `core/heading`
   - `core/list`, `core/list-item`
   - `core/image`, `core/gallery`
   - `core/cover`
   - `core/columns`, `core/column`
   - `core/group`
   - `core/buttons`, `core/button`
   - `core/separator`, `core/spacer`
   - `core/quote`
   - `core/details`
   - `core/embed` (for YouTube/Vimeo URLs explicitly present in source)

   Any other block type (`core/navigation`, `core/template-part`, `core/post-title`, `woocommerce/*`, third-party blocks) is rejected.
   `core/html` / `wp:html` is also rejected; Custom HTML blocks are not a layout or CSS escape hatch.

3. **No template-part references.** Template parts (header, footer, sidebar) are a Site Editor concept resolved against the active theme — they cannot live inside `post_content`. If a section in the screenshot looks like header or footer chrome, skip it; the theme's `parts/header.html` already renders it.

4. **No raw `<script>`, `<iframe>`, `<object>`, `<embed>`, `<form>`.** The pre-skill sanitizer strips these from input. Do not reintroduce them. Use `core/embed` for video URLs that are demonstrably in the source HTML.

5. **No ad hoc inline CSS.** CSS needed to match the source belongs in the active theme's `style.css` or theme.json/block styles. In `post_content`, prefer native block slug attributes and registered className styles:
   - For blocks with native color attrs (`backgroundColor`, `textColor`), use the slug: `"backgroundColor":"surface-raised"`.
   - For source-specific layout details a block cannot express, add a className hook and surface a warning so the theme can add CSS in `style.css`.
   - Hex codes (`#0f4d7a`, `rgb(...)`, `rgba(...)`), `<style>` tags, and inline `style` attributes are rejected.

6. **No hallucinated copy.** Every text node in the output must appear (case-insensitive, whitespace-normalized) as a substring of the source HTML's plain text. The post-skill `output-verify.ts` enforces this.

## Soft rules — best practice, but not auto-enforced

7. **Prefer semantic blocks over custom blocks.** Use existing WordPress core blocks first. If a component cannot be represented with core blocks, use a purpose-built custom block only when the active theme has already registered one for that component; otherwise omit the section and add a warning.

8. **Apply foundation classes via `className`.** When a section maps to a foundation role, set the corresponding style slug:
   - `is-style-soft-card` for raised cards
   - `is-style-accent-primary` for primary-accent backgrounds
   - `is-style-hero` for cover blocks acting as heroes
   - `is-style-display` for display-typography headings
   The list of registered styles lives in the theme's `theme.json` styles section, generated by `replicate-with-blocks`.

9. **Use the `align` attribute consistently.** Full-bleed sections get `align: "full"`, "wide" sections get `align: "wide"`, default content stays unaligned. Mixing alignments produces visual jitter.

10. **Preserve image alt text.** Pull `alt` from the source `<img>` attribute. If empty, leave `alt=""` (decorative); never invent alt text — the verifier rejects it.

11. **Quote attribution stays inside `<cite>`.** Don't promote it to its own paragraph block.

12. **Keep block depth shallow.** Two levels of nesting (`group > columns > column > content`) is normal. Three levels is a code smell. Four levels means the source is mis-classified — re-think the layout.

## Why these constraints

The replica theme's tokens, gradients, and component overrides are wired up via `theme.json` color slugs and registered block-style variants. Inlining a hex breaks that wiring — the user can change the accent color in `theme.json` and the page won't follow. Template parts assume an active theme that registered them; embedded in `post_content`, they crash on theme switch.

The hallucination ban exists for one reason: source content is the contract. The user trusts the migration tool to preserve their copy. A "helpful" rewrite ("we noticed Foo Industries was awkward, here's Foo Co.") is a bug, not a feature. The verifier discards any output containing text that isn't grounded in source.
